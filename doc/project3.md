# 📌 프로젝트 관리 계획서

## 1. 문서 개요

본 문서는 **Mini Drive** 프로젝트 진행을 위한 관리 계획서입니다.  
프로젝트의 **목적, 개발 계획, 팀 구성, 품질 관리, 개발 환경, 산출물 관리, 리스크 관리** 내용을 정리합니다.

---

## 2. 프로젝트 개요

### 프로젝트명

**Mini Drive**

### 프로젝트 목적

Mini Drive는 조직 내 파일을 중앙에서 관리하고 사용자 간 공유를 지원하는 **웹 기반 클라우드 파일 관리 시스템**입니다.  
파일의 버전 관리, 검색, 권한 기반 공유 기능을 제공하여 협업 환경에서 발생하는 비효율 문제를 해결하는 것을 목표로 합니다.  
또한 웹 브라우저를 통해 별도의 설치 없이 접근 가능하도록 설계합니다.

### 주요 기능

| ID | 기능 설명 |
|---|---|
| F01 | 파일 업로드 및 다운로드 |
| F02 | 다양한 파일 형식 지원 |
| F03 | 폴더 생성, 삭제, 이름 변경 |
| F04 | 계층 구조 기반 폴더 관리 |
| F05 | 사용자 간 파일 공유 |
| F06 | 읽기, 수정, 댓글 권한 설정 |
| F07 | 외부 공유 링크 생성 및 만료 설정 |
| F08 | 파일 이름 기반 검색 |
| F09 | 날짜, 사용자, 파일 유형 필터링 |
| F10 | 파일 변경 이력 저장 |
| F11 | 이전 버전 조회 및 복원 |
| F12 | 회원가입 및 로그인 |
| F13 | 사용자 권한 관리 |

## 3. 개발 계획 🛠️

### 개발 절차 모형

요구사항 변경 가능성이 있고, 기능별 구현과 피드백이 반복적으로 필요하기 때문에  
**Agile 방법론 중 Scrum 모델**을 사용합니다.

### 개발 방법론

객체지향 방법론을 사용하여  
파일 관리, 폴더 관리, 공유, 검색, 버전 관리, 사용자 관리 기능을 모듈별로 분석·설계합니다.  
이를 통해 기능 수정, 유지보수, 확장이 쉬운 구조를 목표로 합니다.

### 주요 개발 활동

| 개발 활동 | 설명 |
|---|---|
| 프로젝트 요약 | Mini Drive 프로젝트 요약서 작성 |
| 프로젝트 계획 | 프로젝트 관리 계획서 작성 |
| 요구사항 정의 | 파일 관리, 공유, 검색, 권한 기능 요구사항 정의 |
| 요구사항 분석 | 사용자, 관리자, 개발자 관점의 품질 요구사항 분석 |
| Product Backlog | 구현할 기능을 우선순위대로 정리 |
| Sprint Planning Meeting | 스프린트 시작 시 백로그 선정 및 개발 계획 수립 |
| Daily Scrum Meeting | 매일 진행상황과 이슈 공유 |
| Sprint | 짧은 단위로 기능 구현 및 피드백 진행 |
| Sprint Retrospective | 개선할 점, 잘한 점, 새로 시작할 점 도출 |
| Sprint Review | 스프린트 종료 시 작업 결과 리뷰 및 문서화 |

## 6. 팀 구성 👥

### 팀 구조

본 프로젝트는 팀장의 역할은 존재하지만,  
모든 팀원이 상호 협력하는 **수평적 팀 구조**를 따릅니다.

### 역할 및 책임

| 역할 | 책임 |
|---|---|
| 팀장 | 프로젝트 전반 진행 및 일정 관리 |
| 문서관리 | 프로젝트 계획서, 요구사항 정의서, 설계서 등 산출물 관리 |

| 프론트엔드 | 웹 화면 구성 및 사용자 인터페이스 구현 |
| 백엔드 | 파일 관리, 공유, 검색, 버전 관리 기능 구현 |
| DB 관리 | 사용자, 파일, 폴더, 권한, 버전 정보 데이터베이스 관리 |

## 7. 품질 관리 ✅

### 팀 미팅 계획

| 회의 | 일정 | 주요 내용 | 참석자 |
|---|---|---|---|
| Sprint Planning Meeting | 매주 시작 시 | Sprint 목표 및 Backlog 계획 | 팀원, 팀장 |
| Daily Scrum Meeting | 매일 | 어제 한 일, 오늘 할 일, 이슈 공유 | 팀원, 팀장 |
| Sprint Review | Sprint 종료 시점 | Sprint 기간 동안 구현한 기능 리뷰 | 팀원, 팀장 |
| Sprint Retrospective | Sprint 종료 시점 | 개선할 점, 잘한 점, 새로 시작할 점 도출 | 팀원, 팀장 |
| 비정기 회의 | 필요 시 | 개발 문제, 기능 변경, 오류 발생 시 진행 | 팀원, 팀장, 지도교수 |

### 변경사항 관리

- 기능 변경은 팀원 전체가 권한을 가지며, 다수결의 합의가 필요합니다.
- 변경사항이 발생하는 경우 변경 전후 내용을 반드시 기록합니다.
- 관련 산출물은 팀원이 전부 확인할 수 있도록 공유합니다.
- 의견 충돌로 진행이 멈출 경우 팀장이 더 강한 발언권을 가집니다.
- 변경사항이 결정되면 지도교수에게 반드시 보고합니다.

### 산출물 관리

#### 파일 명명 규칙

| 구분 | 규칙 | 예시 |

| 문서 | `[팀명]_YYMMDD_Doc-문서버전` | `[MiniDrive]프로젝트관리계획서_260426_Doc-001.md` |

| 기타 파일 | 파일명 앞에 날짜 기재, 필요 시 숫자 추가 | 전체 |

## 10. 기타 사항

### 리스크 관리 ⚠️

| No | Risk | 중요도 | 발생 가능성 | 대응 방안 |
|---|---|---|---|---|
| 1 | 파일 업로드 및 다운로드 오류 | High | Medium | 파일 크기 제한, 예외 처리, 업로드 실패 시 재시도 기능을 적용 |
| 2 | 권한 설정 오류 | High | Medium | 읽기, 수정, 댓글 권한을 명확히 분리하고 권한별 테스트 수행 |
| 3 | 파일 검색 성능 저하 | Medium | Medium | 파일명, 날짜, 사용자, 파일 유형 기준 인덱싱 및 필터링 구조 설계 |
| 4 | 파일 손실 | High | Low | 변경 이력 저장, 이전 버전 복원 기능, 백업 정책 적용 |
| 5 | 외부 공유 링크 보안 문제 | High | Medium | 링크 만료 시간 설정, 접근 권한 검증, 인증 절차 적용 |
